<objective>
Close the Phase 05 expansion-scroll gap by separating expansion-driven data refresh from the viewport-reset behavior that currently forces the table back to `scrollTop = 0`.

Purpose: manual verification found that expanding or collapsing a row sends the user back to the top of the table. In the current workspace, `expanded` is included in `serverSideCacheKey`, and a server-side reset effect listens to that key and unconditionally runs `parentRef.current.scrollTop = 0`. That means every expansion change is treated like a fresh spec reset instead of an in-place hierarchy update.

Output: expansion/collapse keeps the user's viewport stable while still invalidating server-side cache and preserving the existing stale-row protections.
</objective>
