## Summary

Phase 3.2 is a small hardening phase with two independent but low-risk fixes.

The first issue is a test collection side effect. `tests/test_frontend_contract.py` imports `app` at module scope (`line 14`), and `dash_presentation/app.py` executes heavyweight module-scope setup during import: it constructs an in-memory adapter (`line 21`) and immediately calls `load_initial_data()` (`line 60`). That function generates a 2,000,000-row Arrow table and pre-materializes hierarchy state (`lines 24-58`). As a result, merely collecting the test module performs expensive data generation and DuckDB initialization, violating QUAL-05.

The second issue is dead exception handling in `update_pivot_table`. Inside that callback, the final execution block has two consecutive `except Exception` clauses (`lines 308-314`). The second can never run because the first already catches the same exception type, so the function currently has unreachable error-handling code. The planner should treat that as a focused cleanup with regression coverage, not a broader callback refactor.

**Primary recommendation:** split the phase into two plans. Plan 01 should remove import-time side effects by making the Dash app import-safe for tests and by moving the test’s app import behind a fixture or test function. Plan 02 should clean `update_pivot_table` down to a single terminal `except Exception` block and verify the suite still collects and runs.

## What The Planner Needs To Know

### Current Root Cause For QUAL-05

`tests/test_frontend_contract.py` currently does two separate things:

1. It defines its own lightweight test fixture via `create_tanstack_adapter(":memory:")` and `create_test_data()` (`lines 17-39`).
2. It also imports `dash_presentation.app` at module scope solely to verify the app object and layout are valid (`line 14`).

That second part is what breaks collection purity. Importing `dash_presentation.app` executes:

- `adapter = create_tanstack_adapter(backend_uri=":memory:")`
- `load_initial_data()`
- 2M-row synthetic data generation
- `adapter.controller.load_data_from_arrow(...)`
- hierarchy materialization

This means the test file is not collection-safe even though its own fixture is otherwise fine.

### Current Root Cause For QUAL-06

Within `update_pivot_table`, exception handling is layered like this:

- one `except Exception` for drill-through failures
- one `except Exception` for cell update failures
- one terminal `except Exception` around the main execute path
- one duplicate terminal `except Exception` immediately after that

Only the final duplicated pair is a defect relative to the requirement. The planner should avoid collapsing the drill-through and cell-update handlers into the outer try unless there is a concrete benefit, because that would widen behavior changes for a phase whose goal is only to remove dead code.

## Recommended Implementation Pattern

### Pattern 1: Make `dash_presentation.app` Import-Safe

The cleanest fix is to stop doing heavyweight work at import time.

Recommended shape:

- keep `app = Dash(__name__)` importable
- move adapter creation and initial data loading behind a helper or factory
- ensure the helper runs only when the app actually needs to serve requests or when the module is executed directly

Two viable approaches:

1. **Minimal lazy bootstrap**
   - Keep global `adapter = None`
   - Add `get_adapter()` that creates the adapter and calls `load_initial_data()` on first use
   - Replace direct `adapter` use inside callbacks with `get_adapter()`
   - Leaves `from dash_presentation.app import app` safe during collection

2. **App factory**
   - Introduce `create_app()` that builds the Dash app and registers callbacks
   - Export `app = create_app()` only after restructuring bootstrap so data loading is opt-in or lazy
   - Better architecture, but slightly broader than this phase needs

**Recommendation:** choose the minimal lazy-bootstrap approach for Phase 3.2. It directly solves QUAL-05 without expanding scope.

### Pattern 2: Move App Import Verification Into A Test, Not Module Scope

Even if the app module becomes lazy, the test should still avoid module-scope imports when the requirement explicitly calls for lazy fixture behavior.

Recommended shape:

- remove `from dash_presentation.app import app` from module scope
- add a small test such as `test_dash_app_import_and_layout_valid()`
- import `app` inside that test or in a fixture
- assert `app.layout is not None`

This keeps collection pure and still preserves the original coverage intent.

### Pattern 3: Surgical Callback Cleanup

For `update_pivot_table`, remove only the unreachable duplicate terminal handler:

```python
    except Exception as e:
        print(f"Error in pivot engine: {e}")
        return [], 0, no_update, no_update, no_update, no_update, no_update
```

The duplicate immediately following it should be deleted. Nothing else in the callback needs structural change to satisfy QUAL-06.

Goal:
- leave `update_pivot_table` with exactly one terminal `except Exception` block

Likely files:
- `dash_presentation/app.py`

Must-haves:
- the duplicated unreachable `except Exception` block is removed
- collection and targeted tests remain green
- no change to callback output arity or fallback semantics

## Concrete Verification Strategy

### For QUAL-05

The key verification is collection behavior, not only runtime behavior.

Recommended checks:

- `pytest tests/test_frontend_contract.py --collect-only -q`
- ensure no simulation-data print output appears
- ensure collection does not require DuckDB bootstrap from `dash_presentation/app.py`

Best automated regression:

- add a test that imports `dash_presentation.app` under `capsys` or `monkeypatch` and asserts no `Generating simulation data` output occurs on import
- or simpler, rely on the module structure change plus `pytest --collect-only` as the acceptance gate for this phase

### For QUAL-06

Recommended checks:

- `rg -n "except Exception" dash_presentation/app.py`
- inspect `update_pivot_table` only, not the whole file
- run the targeted contract test module after cleanup

Best automated regression:

- a small structural test that inspects `inspect.getsource(update_pivot_table)` and asserts the function contains exactly one terminal `except Exception` for the main execution block is possible, but probably unnecessary for this phase

---

## Anti-Patterns To Avoid

- Do not solve QUAL-05 by deleting the app import check entirely. Preserve the original intent of verifying the Dash app still imports and has a layout.
- Do not keep module-scope adapter initialization and try to hide the side effect with test monkeypatching. The production module should become import-safe.
- Do not broaden Phase 3.2 into a full app-factory rewrite unless the minimal lazy-bootstrap path fails.
- Do not refactor all callback error handling into one giant try/except. QUAL-06 only requires removal of dead duplicate handling, not a behavioral rewrite.

## Open Questions

1. Should the lazy bootstrap happen on first callback execution only, or also when `app.layout` is accessed?
   - Recommendation: first callback execution is sufficient for this phase. `app.layout` can remain static and empty-data-safe.

2. Is the `print("Generating simulation data (2M rows)...")` output relied on by any manual workflow?
   - Nothing in the repo suggests it is. Treat it as debugging noise, not required behavior.

3. Should the test file continue carrying its own `sys.path.append(...)` lines?
   - Probably yes for now. They are not the phase target, and `conftest.py` already exists from prior work. Avoid scope creep unless they break verification.
