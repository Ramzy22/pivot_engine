## Summary

This phase has two tightly coupled goals: add toggleable structured logging to the adapter layer so request payloads and response totals can be inspected at runtime, then use that visibility to confirm and fix the duplicate grand total row.

The duplicate grand total is caused by two independent injection paths converging. The **backend** (`controller.py _append_totals_row`) appends a totals row whose dimension columns are all `None` (row-level) to every response when `spec.totals=True`. The **adapter** (`tanstack_adapter.py handle_hierarchical_request`) then traverses the hierarchy result and detects the same row at `depth==0` with the first dimension column `None`, marking it `_id='Grand Total'`. The frontend (`DashTanstackPivot.react.js tableState`) detects any row matching `_isTotal==true` or `_path=='__grand_total__'` and pushes its id into `rowPinning.bottom` — which causes TanStack Table to render the row a second time in the pinned-bottom area while also keeping it in the regular row list. The `seen_grand_totals` deduplication set in `handle_hierarchical_request` (lines 498-529) is keyed on a measures-value tuple that can produce false-negative matches (collision when two rows have identical measure values), so it does not reliably suppress the second copy.

The fix strategy is: guarantee the backend emits the grand total row **exactly once** per response, regardless of which code path produced it, then add a regression test that asserts count-of-`_isTotal`-rows equals exactly 1 when totals are requested. The debug logging feature must be implemented first so the planner has observable evidence of the double emission before touching controller logic.

**Primary recommendation:** Add an `AdapterDebugLogger` (a thin wrapper on Python's stdlib `logging`) to `tanstack_adapter.py`, controlled by a `debug` flag on `TanStackPivotAdapter.__init__`. Then fix the root cause in `controller.py _append_totals_row` and `tanstack_adapter.py handle_hierarchical_request` to enforce single-total semantics before returning from the adapter.

### Pattern 1: Toggleable Debug Logger on Adapter

**What:** A small helper class or inline function that logs when `self._debug` is `True`.
**When to use:** Wraps the two key moments: (a) when `handle_hierarchical_request` or `handle_virtual_scroll_request` receives a request, (b) when `convert_pivot_result_to_tanstack_format` is about to return rows.

**Example:**
```python
# In tanstack_adapter.py
import logging

_adapter_logger = logging.getLogger("pivot_engine.adapter")

class TanStackPivotAdapter:
    def __init__(self, controller, debug: bool = False):
        self.controller = controller
        self.hierarchy_state = {}
        self._debug = debug

    def _log_request(self, method: str, request: "TanStackRequest", **extra):
        if not self._debug:
            return
        _adapter_logger.debug(
            "adapter_request",
            extra={
                "method": method,
                "table": request.table,
                "grouping": request.grouping,
                "totals": request.totals,
                "filters_keys": list((request.filters or {}).keys()),
                **extra,
            }
        )

    def _log_response(self, method: str, rows: list):
        if not self._debug:
            return
        total_rows = [r for r in rows if r.get("_isTotal") or r.get("_path") == "__grand_total__"]
        _adapter_logger.debug(
            "adapter_response",
            extra={
                "method": method,
                "row_count": len(rows),
                "total_row_count": len(total_rows),
                "has_grand_total": any(r.get("_id") == "Grand Total" for r in rows),
            }
        )
```

**Enabling in app.py:**

### Pattern 2: Single-Total Enforcement in `_append_totals_row`

**What:** Before appending the totals row, check whether the existing table already contains a row whose dimension columns are all `None` (the signature of a grand total from ROLLUP/grouping_sets). If so, skip appending a second.

**Why it matters:** `_execute_standard_pivot_async` calls `_append_totals_row` unconditionally whenever `metadata['needs_totals']` is True. If the underlying Ibis query already produced a ROLLUP grand total row (all dims=None), calling `_append_totals_row` appends a duplicate.

**Example:**
```python
def _append_totals_row(self, table, spec, metadata, column_values=None):
    # Guard: if a grand total row already exists in the table, do not append another
    if self._table_already_has_grand_total(table, spec):
        return table
    totals_query = self._build_totals_query(spec, column_values=column_values)
    ...

def _table_already_has_grand_total(self, table: "pa.Table", spec) -> bool:
    """Return True if the table already contains a row where all row-dimension columns are None."""
    import pyarrow.compute as pc
    row_dims = getattr(spec, "rows", [])
    if not row_dims:
        return False
    for col_name in row_dims:
        if col_name not in table.column_names:
            return False
    # A grand total row has all dimension values as None
    col = table.column(row_dims[0])
    null_mask = pc.is_null(col)
    return pc.any(null_mask).as_py()
```

### Pattern 3: Reliable Grand Total Dedup in `handle_hierarchical_request`

**What:** Replace the fragile measure-value-tuple keying in `seen_grand_totals` with a structural check: a node at `depth==0` where the first row-dimension column is `None` is a grand total. Allow at most one such node through.

**Current fragile code (lines 498-529 of tanstack_adapter.py):**
```python
is_grand_total = (current_depth == 0 and node.get('region') is None)  # HARDCODED field name!
```

The `region` check is hardcoded. It should use `pivot_spec.rows[0]` instead.

**Fixed approach:**
```python
# At traversal time, use a boolean flag instead of a set
grand_total_emitted = False

def traverse(parent_key):
    nonlocal grand_total_emitted
    nodes = hierarchy_result.get(parent_key, [])
    current_depth = len(parent_key.split("|||")) if parent_key else 0

    for node in nodes:
        if not isinstance(node, dict):
            continue

        # Detect grand total: depth==0 and first dimension is None
        first_dim = pivot_spec.rows[0] if pivot_spec.rows else None
        is_grand_total = (
            current_depth == 0
            and first_dim is not None
            and node.get(first_dim) is None
        )

        if is_grand_total:
            if grand_total_emitted:
                continue  # skip duplicate
            grand_total_emitted = True
        ...
```

### Anti-Patterns to Avoid

- **Hardcoding field names in grand total detection:** The current `node.get('region') is None` check in `handle_hierarchical_request` breaks for any dataset where the first row dimension is not `region`. Always use `pivot_spec.rows[0]`.
- **Keying dedup on measure values:** If two different rows happen to have the same measure values, the set-based dedup skips legitimate data rows. Use structural detection (dimension is None at depth 0) instead.
- **Filtering on the frontend:** The frontend comment at line 1526 says "Do not filter server-side data on client to preserve index alignment with virtualizer." Never try to remove the duplicate in `DashTanstackPivot.react.js` — the fix belongs in Python.
- **Adding the debug flag to `create_tanstack_adapter` only:** The `app.py` also calls `create_tanstack_adapter()` directly; the factory function must accept and pass through the `debug` flag.

## Common Pitfalls

### Pitfall 1: Debug Flag Not Threaded to Factory

**What goes wrong:** `create_tanstack_adapter()` is called in `app.py` without a `debug` parameter, so even after adding `debug` to `TanStackPivotAdapter.__init__`, the factory silently ignores it.
**Why it happens:** The factory function at the bottom of `tanstack_adapter.py` constructs the adapter without forwarding kwargs.
**How to avoid:** Update `create_tanstack_adapter` signature to `def create_tanstack_adapter(backend_uri=":memory:", debug=False)` and pass `debug=debug` to the `TanStackPivotAdapter(controller, debug=debug)` call.
**Warning signs:** Logs never appear even when `debug=True` is passed to `create_tanstack_adapter`.

### Pitfall 2: `_append_totals_row` Called Before Hierarchy Traversal

**What goes wrong:** The totals row is appended at the controller level (in `_execute_standard_pivot_async`) to the raw query result table. Then the adapter's `handle_hierarchical_request` traversal iterates over that table (after `run_hierarchical_pivot_batch_load`) and detects the same row again.
**Why it happens:** The controller's `run_pivot_async` adds a totals row unconditionally; `run_hierarchical_pivot_batch_load` calls `run_pivot_async` internally for each batch. The totals row travels through batch result → hierarchy dict → traversal → visible_rows.
**How to avoid:** The guard in `_append_totals_row` (Pattern 2 above) prevents appending a second copy at the controller level. Additionally, the traversal's grand total dedup (Pattern 3) catches any that slip through.
**Warning signs:** After the controller fix, the regression test still fails — that means the duplication is coming from the traversal path, not the append path.

### Pitfall 3: `seen_grand_totals` Set Uses Mutable Data

**What goes wrong:** The current `measures_key = tuple(sorted([(k, v) for k, v in node.items() if k not in [...]]))` includes float values that may be `None` or `NaN`, making the tuple hashable but unreliable.
**Why it happens:** When two regions have the same measure total, the set treats them as the same grand total and skips the second, corrupting legitimate data rows.
**How to avoid:** Never use measure values to identify grand total rows. Use `first_dim is None at depth 0` as the sole discriminator (Pattern 3 above).
**Warning signs:** One of the top-level dimension rows (not the grand total) goes missing from the response.

### Pitfall 4: PyArrow Null Check on Non-Nullable Column

**What goes wrong:** If the first dimension column was loaded as `pa.string()` non-nullable (no nulls at load time), `pc.is_null` returns an all-False array, so the guard concludes "no grand total exists" and appends a duplicate.
**Why it happens:** PyArrow columns from DuckDB GROUP BY results always include a NULL row for ROLLUP totals, but the schema may declare the field non-nullable.
**How to avoid:** Cast the column to nullable before null-checking, or catch the `ArrowInvalid` exception and return `False` from the guard.

### Pitfall 5: Regression Test Uses In-Memory DuckDB But Does Not Reset

**What goes wrong:** If the regression test reuses the module-scope adapter fixture from `test_frontend_contract.py`, it inherits the module-level `from dash_presentation.app import app` which triggers `load_initial_data()` and a 2M-row table load.
**Why it happens:** `test_frontend_contract.py` imports `app` at module scope (the existing QUAL-05 issue being fixed in Phase 3.2).
**How to avoid:** Put the grand total regression test in a new file (`tests/test_grand_total_single.py`) that creates its own adapter with a small synthetic dataset, with no dependency on `dash_presentation.app`.

def _append_totals_row(self, table, spec, metadata, column_values=None):
    totals_query = self._build_totals_query(spec, column_values=column_values)
    if totals_query is None:
        return self._compute_totals_arrow(table, metadata)
    try:
        totals_table = totals_query.to_pyarrow()

        return self._compute_totals_arrow(table, metadata)
    aligned_totals = self._align_total_row_to_result(totals_table, table)
    return pa.concat_tables([table, aligned_totals])

## Open Questions

1. **Does `run_hierarchical_pivot_batch_load` internally call `run_pivot_async` (which appends totals) for each batch?**
   - What we know: `run_hierarchical_pivot_batch_load` is defined on `ScalablePivotController` and calls down to the hierarchy/materialization managers.
   - What's unclear: Whether each batch result goes through `_execute_standard_pivot_async` (which calls `_append_totals_row`) or bypasses it.
   - Recommendation: During Wave 1 (debug instrumentation), enable `debug=True` on the adapter and add a `_log_request` call inside `_append_totals_row` to count invocations per request. This will confirm whether the duplicate originates in the controller's append path or the traversal path.

2. **Does the frontend `rowPinning.bottom` cause a visual double render even when the data contains exactly one `_isTotal` row?**
   - What we know: TanStack Table's `rowPinning` renders pinned rows in a separate pinned area AND keeps them in the main row list by default (unless `keepPinnedRows: false` is set).
   - What's unclear: Whether the current TanStack Table v8 version used in this project has `keepPinnedRows` option and whether it is set.
   - Recommendation: The backend fix (single total row in response) is necessary but may not be sufficient if `rowPinning` duplicates the display. Add a follow-up check: after the backend fix, if the row still shows twice, set `keepPinnedRows: false` in the TanStack table options in `DashTanstackPivot.react.js`.

3. **Is `spec.totals` correctly passed through `handle_hierarchical_request`?**
   - What we know: `convert_tanstack_request_to_pivot_spec` maps `request.totals` to `spec.totals` (line 158 of tanstack_adapter.py). The Dash app passes `totals=show_col_totals` from the `showColTotals` prop.
   - What's unclear: If `showColTotals` defaults to `True` in the React state, totals are always requested even when the user hasn't explicitly asked for them.
   - Recommendation: Treat this as a known behavior, not a bug. The fix must work correctly when `totals=True`.
