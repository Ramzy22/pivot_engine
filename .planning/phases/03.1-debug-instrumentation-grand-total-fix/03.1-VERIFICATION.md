### Human Verification Required

#### 1. Grand total visual position in the UI

**Test:** Open the pivot table in a browser with a dataset that has multiple row dimensions. Verify the grand total row appears at the bottom (after all data rows), not duplicated at the top overlapping a data row.
**Expected:** A single grand total row at the bottom of the table, no phantom row at top.
**Why human:** The `_dedup_grand_total` helper preserves the first occurrence, not the last. If the upstream query emits the grand total row first (before data rows), it will appear at the top. This positional correctness cannot be determined from unit tests alone.

**Test:** Enable `debug=True` on the adapter and process a request. Inspect the logged records to confirm they contain meaningful request and response data (table name, grouping, row count, total row count).

**Why human:** The unit test only asserts a record was emitted from `pivot_engine.adapter` — it does not validate the record's extra fields (they are passed via `extra={}` to the logger and depend on the handler configuration).

## Summary

Phase 3.1 achieved its goal. BUG-14 is resolved: the duplicate grand total row is eliminated at three layers of defense:

1. **Traversal layer** (`traverse()` in `handle_hierarchical_request`): A `grand_total_emitted` boolean flag using `pivot_spec.rows[0]` for generalized detection replaces the fragile `seen_grand_totals` set with hardcoded `'region'` field name.

2. **Controller layer** (`_append_totals_row` in `controller.py`): The `_table_already_has_grand_total` guard detects ROLLUP-produced grand total rows (via `pc.is_null` on the first row dimension) and returns the table unchanged if one already exists.

3. **Post-processing layer** (`handle_virtual_scroll_request`): `_dedup_grand_total` is applied unconditionally to all three return paths, providing a safety net regardless of which internal code path produces the rows.

The debug logging facility (`debug=True` on adapter creation) is functional, emitting records to `logging.getLogger("pivot_engine.adapter")` from both `_log_request` and `_log_response`. All 4 regression tests pass GREEN; the full 20-test suite has no regressions.

One minor human verification is recommended: confirm the grand total row appears at the correct vertical position in the UI (bottom, not top), since `_dedup_grand_total` preserves first occurrence and upstream ordering determines position.
